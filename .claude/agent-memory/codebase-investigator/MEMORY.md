# Codebase Investigator Memory

## Project Structure
- Monorepo: packages/core, packages/codegen, packages/typegen, packages/builder, packages/formatter, packages/tsc, packages/lsp, packages/vscode-extension
- Playgrounds: vite-react, expo-metro, nextjs-webpack, nestjs-compiler-tsc, hasura
- fixture-catalog: Integration test fixture with real generated files

## Prebuilt Type System (Investigated 2026-03-05)
See: `prebuilt-type-system.md` for details.

Key gap: `PrebuiltTypeRegistry` in `packages/core/src/prebuilt/types.ts` is a minimum-viable constraint
but the generated `PrebuiltTypes_*` shapes have `typename` on fragments (not in the registry type)
and `void` for fragment input (not assignable to `object` on operations, though `unknown` covers `void`).
The registry type is imported but never used as a `satisfies` constraint in generated files.

## Testing Conventions
- Unit tests colocated with source: `src/**/*.test.ts`
- Type tests: `packages/core/test/types/*.test.ts` (use Bun test runner with type assertions)
- Integration/E2E: `packages/typegen/test/e2e/`
- All test runs via `bun run test`

## Build/Codegen Output Structure
- `_internal.ts` — main schema runtime
- `_internal-injects.ts` — scalar imports
- `_defs/` — chunked type definitions
- `index.ts` — generated prebuilt-typed wrapper (generated by `generateIndexModule`)
- `types.prebuilt.ts` — generated by typegen `emitPrebuiltTypes`, or stub from codegen

## Runtime Behavior Inconsistencies (Investigated 2026-03-08)
See: `runtime-behavior-report.md` for full details.

Key findings:
1. Field-level directives silently dropped in tagged templates (`fragment-tagged-template.ts:111`)
2. Fragment spread `path` always `null` in tagged templates vs callback builder tracking path
3. `createOperationTaggedTemplate` missing schema root null-check (unlike callback/compat builders)
4. `operationRegistry` contaminated across builder integration tests (no `__resetRuntimeRegistry`)
5. `cachedGql` module-level singleton in `evaluation.ts` not reset in integration tests
6. Fragment metadata callback semantics differ between syntaxes

## Type Inconsistencies (Investigated 2026-03-08)
See: `type-inconsistencies-report.md` for full findings.
Key: `IntermediateElements` CanonicalId brand stripped at API boundary; `FieldSelectionData.variableDefinitions` diverges between fragments (VariableDefinitions) and operations (VariableDefinitionNode[]); `AnyGqlContext.compat` wrong return type; `rootTypeName ?? undefined` runtime throw risk in typegen emitter.

## Cross-Package Inconsistencies (Investigated 2026-03-08)
See: `cross-package-inconsistencies.md` for full details.

Key findings:
1. `tsconfig.editor.json` references incomplete: builder (missing config), codegen (missing config), cli (missing config/typegen/common), formatter (missing common), sdk (missing codegen/common)
2. `@soda-gql/swc` duplicates `normalizePath` locally without `normalize()` call unlike `@soda-gql/common`
3. `@swc/core` version ranges: `^1.0.0` vs `^1.6.3` vs `^1.10.0`
4. `neverthrow`: `^8.2.0` in config, `^8.1.1` everywhere else
5. `colocation-tools/tsconfig.editor.json` has stale outDir `helpers-internal` (old package name)
6. `cli` package: only `require`/`default` conditions, no `import` condition
7. `prettier` devDep in `@soda-gql/swc` appears unused

## API Contract Inconsistencies (Investigated 2026-03-08)
See: `api-contract-report.md` for full details.

Key critical/high findings:
- `config.codegen.chunkSize` silently dropped by CLI (`schemaCommand`) and SDK (`codegenAsync`) — never forwarded to `runCodegen`
- String form of `inject` config always sets `adapter` = same path as `scalars`, even if the file doesn't export an adapter
- `TypegenOptions` exported from `@soda-gql/typegen` is a dead export — `runTypegen` uses `RunTypegenOptions` (different shape)
- `ArtifactLoadError`: wrong code `ARTIFACT_NOT_FOUND` used for I/O read failures (permission errors, etc.)
- `FieldSelectionData.variableDefinitions` is `VariableDefinitions` (VarSpecifier map) for fragments vs `VariableDefinitionNode[]` for operations
- `toImportSpecifier` duplicated between `codegen/runner.ts` and `typegen/runner.ts` with subtle drift in edge case behavior
- `@soda-gql/common/test` subpath only has `@soda-gql` condition — not accessible to external consumers at runtime

## Error Handling Inconsistencies (Investigated 2026-03-08)
See: `error-handling-report.md` for full details.

Key findings:
- No `fromPromise` / `_unsafeUnwrap` in production source (only in tests — correct)
- 3 CRITICAL: `builder-session.ts` generator throws BuilderError caught by outer try-catch; `registry.ts` throws BuilderError from evaluation; `evaluation.ts` throws BuilderError from non-Composer helper
- 3 HIGH (+1): `plugin/session.ts` throws plain Error losing BuilderError structure; `codegen/graphql.ts:187` throws inside Result-returning function; `sandbox.ts` throws from VM require; `format.ts:183` unawaited async write
- `core/src/graphql/result.ts` has a custom Result type (not neverthrow) — API incompatible with neverthrow Result
- 4 different `assertUnreachable` implementations with inconsistent message formats
- `schema-loader.ts` uses CONFIG_* error codes for non-config runtime artifacts
